What this app is

ClaudePod is a mobile-first PWA for long-form voice conversations with Claude — the Claude chat experience, but you listen instead of read. The target use case is asking a question at the gym and listening to a multi-minute answer between sets. Latency is explicitly not a priority; quality and reliability are.

The voice pipeline (read this before touching anything)

1. Outbound: question → spoken answer

  1. messages.send (mutation) inserts the user message and an assistant placeholder (status: "pending"), then schedules internal.ai.generate.
  2. ai.generate (node action) streams from Claude (client.messages.stream, model CLAUDE_MODEL, system prompt SPOKEN_SYSTEM_PROMPT, adaptive thinking). Streamed text is patched into the message document (throttled to ~350 ms). Convex reactivity is the streaming transport — the client just subscribes to messages.list; there is no SSE/WebSocket of our own.
  3. In the same action, a SentenceChunker cuts the stream at sentence boundaries (first chunk small for early playback, later chunks larger for prosody). Each chunk becomes an audioSegments row (pending → ready) whose mp3 lives in Convex file storage. Synthesis runs on a sequential promise queue concurrently with the Claude stream, so early sentences are playable while later ones are still being generated.
  4. TTS goes through getTtsProvider() (convex/tts/index.ts). Adding a provider = implement TtsProvider, register it there, document the env var. Never call a TTS HTTP API from anywhere else.
  5. Request stitching: chunks of one message are separate TTS requests, and unconditioned requests each get an independent loudness/prosody “take” — audible as a volume jump at every chunk boundary. So each synthesize() call receives a SynthesisContext (previous chunk's text + up to 3 prior request-ids, threaded through the sequential queue by makeStitchTracker in ai.ts). ElevenLabs maps these to previous_text/previous_request_ids and retries once unstitched if a stitched request fails (expired ids, plan limits); OpenAI ignores the context.

2. Playback: segments → one continuous chapter

src/lib/player.tsx owns a single app-wide HTMLAudioElement — required for iOS Safari, where audio must start from a user gesture. unlock() plays a base64 silent mp3 inside any gesture (mic toggle, send tap, play tap); after that the element can be re-pointed at segment URLs programmatically (this is how auto-play of fresh answers works). Starting/stopping mic capture makes the OS (iOS especially) reconfigure the audio session, which can pause or resume the element behind React's back — useMic reports capture changes via onCaptureChange, wired to player.resync(), which re-asserts the intended play/pause state (with delayed retries, since the session change can land after the toggle). To stop iOS from also swinging output loudness between the media and voice-processed routes on every mic toggle, src/main.tsx pins navigator.audioSession.type = "play-and-record" at startup (Audio Session API, Safari 17+; harmless no-op elsewhere) so the session category — and therefore volume — stays constant whether or not the mic is capturing.

The ordered segments of a message are presented as one virtual timeline: elapsed = sum of prior segment durations + element currentTime. Seeking and ±15/30 s skips map a global offset back to (segment, offset). Durations start as server estimates (ElevenLabs: exact from CBR byte length; otherwise ~14 chars/sec) and are refined to exact values from loadedmetadata via audio.setSegmentDuration. If playback reaches a segment whose TTS isn't done, the player shows “buffering” and resumes reactively when the row turns ready; errored segments are skipped, never fatal. The first chunk is cut at the first sentence boundary (firstChunkMinChars) so audio starts as soon as any speakable sentence exists.

Playback UI: there is exactly one control surface — the NowPlayingBar docked above the mic/composer (play/pause, ±15/30 s, speed, scrubber, elapsed/remaining). Assistant responses render as flowing text (AssistantMessage): each TTS sentence-chunk is a tappable span — tapping seeks playback to that chunk (player.seekToSegment) — and the chunk currently being spoken is highlighted karaoke-style via player.activeSegmentIndex. Each response keeps one small play/listen button in its header row. Do not add per-message scrubbers/speed controls — speed and position are global state and live only in the bar.

3. Inbound: microphone rules (the part that makes it gym-proof)

src/lib/useMic.ts implements two strict modes:

StateBehavior
Claude audio playing + mic on Keep the Deepgram socket open but act on nothing except a loose “stop” (STOP_COMMAND_REGEX in appConfig.ts: “stop”, “okay stop”, “claude stop”…), which pauses playback. Gym chatter, music, grunts are discarded. No barge-in on arbitrary speech — ever. There is deliberately no “say stop to pause” banner. Instead, everything heard during playback renders as a live transcript in MicDock — feedback for why a mangled “stop” didn't take — and non-matching words fade ~2 s after they finalize without ever touching the prompt draft.
Playback stopped + mic on Finalized transcripts accumulate into a draft. After silenceMs of quiet (user-tunable, default 2750 ms, Settings slider 1.5–5 s) the draft is submitted. The UI shows the live transcript and a “Sending in Ns…” countdown with Cancel / Edit (Edit moves the draft into the text composer). Any new speech resets the countdown.

Auth: the browser opens wss://api.deepgram.com/v1/listen with a short-lived token minted by stt.getDeepgramToken (Deepgram /v1/auth/grant, 300 s TTL) passed as the ["bearer", token] WebSocket subprotocol. The real DEEPGRAM_API_KEY never reaches the client. Sockets auto-reconnect with a fresh token while the mic stays on. Audio is MediaRecorder audio/webm;codecs=opus in 250 ms slices.

4. Resume / regenerate

Conversations are fully persisted; reopening one finds its audio intact (storage URLs are re-signed by the audio.getSegments query). If a completed message has no playable audio (historical failure, provider switch), ChapterPlayer offers “regenerate”, which calls ai.regenerateAudio — it re-chunks the stored text and synthesizes with the current provider.

Conventions

  • Model id lives in exactly one place: CLAUDE_MODEL in convex/appConfig.ts. Same for voices, chunk sizes, defaults — config goes in appConfig.ts, not inline.
  • Claude must speak, not write: any prompt change must preserve the no-markdown / no-lists / no-code spoken-prose contract in convex/prompts.ts — the output is piped straight into TTS.
  • Secrets live on the Convex deployment (env vars), synced from .env by bun run sync-env. The browser bundle must never contain an API key; the only key-shaped thing the client touches is the short-lived Deepgram token.
  • Ownership checks: every public query/mutation/action resolves getAuthUserId and verifies the row's userId. Internal functions trust their callers.
  • iOS audio: never create a second Audio/“AudioContext”; route all playback through the player provider so the one unlocked element is reused. Call player.unlock() inside any new user-gesture handler that may lead to playback.
  • Mobile-first: inputs ≥16px (no zoom-on-focus), safe-area insets on header/dock, thumb-reachable primary controls.

Environment & setup

bun install
bunx convex dev          # provisions/links the Convex dev deployment (.env.local)
bun run setup-auth       # one-time: JWT keys + SITE_URL for Convex Auth
cp .env.example .env     # fill in ANTHROPIC_API_KEY, DEEPGRAM_API_KEY, ELEVENLABS_API_KEY
bun run sync-env         # pushes those keys to the Convex deployment
bun run dev              # vite on :5173  (run `bun run dev:backend` alongside)

Switching TTS to the cheaper fallback: set TTS_PROVIDER=openai (and OPENAI_API_KEY) in .env, re-run bun run sync-env. New audio uses OpenAI; old cached audio keeps playing; “regenerate” re-voices old messages.

Production

The app is deployed at https://claudepod.vercel.app (Vercel project claudepod, scope johnqs-projects), backed by the production Convex deployment polished-opossum-478.convex.cloud — a separate deployment from dev, with its own data, env vars, and auth keys (accounts do not carry over from dev). Deploys are continuous: the Vercel project is connected to the GitHub repo (Xyzrr/claudepod), so every push to main deploys to production and every PR gets a preview. The build command lives in vercel.json: on production it runs npx convex deploy --cmd 'npm run build' (pushes Convex prod functions and builds the frontend against the prod URL, authenticated by the CONVEX_DEPLOY_KEY env var on Vercel, production scope); on previews it builds the frontend only, pointed at prod Convex via a preview-scoped VITE_CONVEX_URL. vercel.json also holds the SPA fallback rewrite. bun run deploy (scripts/deploy.ts) remains as a manual escape hatch — it ships a locally-built bundle directly and needs VERCEL_TOKEN in .env. If prod env vars change, use bun run sync-env -- --prod / bun run setup-auth -- --prod. The dev tunnel alternative for phone testing is cloudflared tunnel --url http://localhost:5173 (mic requires HTTPS).

Testing notes

  • bunx tsc --noEmit and bunx vite build must stay green.
  • The mic pipeline needs a real device/mic; on desktop Chrome you can fake it with --use-fake-device-for-media-stream.
  • iOS-specific behaviors to re-test after player changes: first-gesture unlock, lock-screen MediaSession controls, playback continuing with screen off.
